================================================================================
1. ROLE / SESSION AUTHORIZATION (CRITICAL)
================================================================================

BEFORE:
  - getUserProfile() trusted aj_profile cookie for role (cookie was not httpOnly).
  - Any logged-in user could set cookie role to "admin" in browser DevTools.
  - requireAdminApiSession / requireStaffApiSession used cookie-trusted role.
  - set-session accepted client-supplied profile.role from request body.
  - set-session did not verify JWT before writing cookies.

AFTER:
  - loadAuthorizedProfile() always reads role from public.profiles via DB.
  - verifySessionRole() uses supabase.auth.getUser() + DB role check.
  - getUserProfile() uses loadAuthorizedProfile() only (no cookie role trust).
  - set-session verifies access_token via getUser() before any cookie write.
  - set-session requires access_token + refresh_token (no unverified mirror path).
  - aj_profile cookie is httpOnly and set only after token verification.
  - requirePortalMemberApiSession now uses verifySessionRole (DB role).

Files:
  lib/security/auth/loadAuthorizedProfile.ts
  lib/security/auth/verifySessionRole.ts
  lib/security/auth/requireAdminApi.ts
  lib/security/auth/requireStaffApi.ts
  lib/auth/getUserProfile.ts (updated)
  app/api/auth/set-session/route.ts (updated)

Import from: @/lib/security

2. OPEN API ENDPOINTS LOCKED DOWN
================================================================================

BEFORE:
  - POST /api/notifications/counselling-scheduled — no auth (open email relay).
  - POST /api/auth/load-profile — unauthenticated userId + service-role fallback.
  - GET  /api/health/supabase — public config disclosure.
  - GET  /api/health/supabase-admin — public; called Auth listUsers.

AFTER:
  - counselling-scheduled — requireStaffApiSession (admin/super_admin/employee).
  - load-profile — requires valid session OR accessToken only (no bare userId).
  - health/supabase + health/supabase-admin — admin-only always.
  - auth/callback — safeRelativePath blocks open redirects (//evil.com).

Files:
  app/api/notifications/counselling-scheduled/route.ts
  app/api/auth/load-profile/route.ts
  app/api/health/supabase/route.ts
  app/api/health/supabase-admin/route.ts

3. RATE LIMITING
================================================================================

BEFORE:
  - No app-level rate limits on login, password bootstrap, or email APIs.

AFTER:
  - IP-based in-memory rate limits via lib/security/rateLimit.ts
  - Returns HTTP 429 with Retry-After when exceeded.

Limits applied:
  auth:sign-in           15 / 15 min
  auth:first-login       10 / 15 min
  auth:needs-first-login 20 / 15 min
  auth:set-session       30 / 1 min
  auth:load-profile      30 / 1 min
  email:outreach         30 / 1 min
  email:counselling      20 / 1 min

Files:
  lib/security/rateLimit.ts

4. DATABASE RLS (SUPABASE — RUN IN SQL EDITOR)
================================================================================

BEFORE:
  - profiles_rls_fix.sql allowed ANY authenticated user to SELECT all profiles
    (policy: using (true)).

AFTER:
  - profiles_rls_tighten.sql replaces broad read with scoped policies:
      * Self read (id = auth.uid())
      * Admin read all (is_admin())
      * Mentor read students in department / assigned
      * Student read assigned mentor
      * Employee read their manager (employee_details)

ACTION REQUIRED:
  Run AJ_Academy_SB/profiles_rls_tighten.sql in Supabase (after profiles_rls_fix.sql).

================================================================================

AFTER:
