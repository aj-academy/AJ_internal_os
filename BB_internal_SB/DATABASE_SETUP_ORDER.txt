Apply these SQL scripts in Supabase SQL Editor (same production project as the app).
Match the NEXT_PUBLIC_SUPABASE_URL in Vercel / .env.local.

Order (core → features → RLS):

1) schema.sql
   Profiles, employee_details, audit_logs, helpers (get_user_role, is_admin).

ATTENDANCE (employee + admin attendance + work summaries) — pick ONE approach:

  A) RECOMMENDED — attendance_module.sql (one paste: helpers + tables + RLS for this domain).
  B) LEGACY — same content in three steps: attendance_rls_prereqs.sql (if helpers missing),
     then attendance_schema.sql, then attendance_rls.sql.

2) attendance_schema.sql  (only if you did not use attendance_module.sql)
   attendance_records, work_summaries, work_from_home_requests, leave_requests, attendance_settings.
   Safe to re-run: adds missing columns (e.g. work_summaries.manager_remarks) via ALTER … IF NOT EXISTS.
   NOTE: This file defines a simplified permission_requests. If you use permission_requests_schema.sql
   instead (recommended), SKIP creating permission_requests from attendance_schema.sql or drop the weak
   table first — prefer permission_requests_schema.sql for approval fields used by Admin Attendance UI.

3) permission_requests_schema.sql
   Full permission_requests (approved_by, approved_at, rejection_reason, permission_type, updated_at).

4) employee_master_schema.sql
   Optional: departments / designations / employee_master extensions.

5) client_lead_schema.sql
   Single public.clients row model (leads + converted clients), full CRM columns, RLS, realtime on clients.
   This is the authoritative clients table — not duplicated by the next script.

5b) client_master_schema.sql
   Run after (5). Satellite tables: lead_followups, lead_activities, client_documents (+ RLS + realtime).
   Plus idempotent ALTER/legacy constraint drops for databases that were created from an older narrow clients table.

6) task_schema.sql
   public.tasks + RLS (admin CRUD; employee read/update own; employee insert for self / same department / shared project team via tasks_employee_may_assign_to;
   get_team_assignees() RPC (employee assignee picker); get_task_assignees() RPC (admin/manager assignee picker); manager read/insert/update/delete-all) + optional realtime.
   Re-run this file after updates so new policies and RPCs stay in sync (safe when using DROP POLICY IF EXISTS / CREATE OR REPLACE).

6a) task_notifications_columns.sql
   Run AFTER (6). Adds tasks.assigned_by (who assigned / last reassigned) and tasks.completion_summary (assignee text on completion) for email flows.

6b) project_master_schema.sql
   Run AFTER (6a). Adds public.projects, project_activities, project_team_members; tasks.project_id FK;
   triggers to sync project progress from tasks; RLS for projects/activities/team; extends task SELECT for project team.
   Re-run this script after pulling updates if RLS helper functions / policies change (uses OR REPLACE / DROP POLICY IF EXISTS).

6c) finance_schema.sql
   Run AFTER (6b). finance_transactions, expense_claims, project_payments, finance_categories, finance_activities;
   triggers refresh projects.advance_paid/pending_amount from project_payments; approved claims post expense rows; RLS for roles.

6d) system_settings_schema.sql
   Run AFTER (6c) or anytime after profiles. Key/value JSON store for Admin System Settings (company, attendance, CRM lists, etc.).

7) company_policies_schema.sql
   company_policies + policy_acceptances (Employee policy gate + Admin policy URLs).

8) rls-policies.sql
   Core RLS on profiles / employee_details / audit_logs / settings.

9) attendance_rls.sql
   Attendance + work_summaries + permission_requests + profiles read policies used by Employee/Admin attendance.
   Skip if you already ran attendance_module.sql (Section C is the same RLS).
   If the editor errors with "function public.is_admin() does not exist", run attendance_rls_prereqs.sql
   first (or apply schema.sql step 1 so get_user_role + is_admin exist), then re-run attendance_rls.sql.
   If PostgREST returns "infinite recursion detected in policy for relation profiles", run
   fix_get_user_role_bypass_rls.sql (get_user_role must use SET row_security = off), then retry login.

Re-run scripts are generally safe where they use CREATE IF NOT EXISTS / DROP POLICY IF EXISTS.
